# GoSend 开发计划

## 基线与原则

- 协议基线：LocalSend Protocol v2.1；设备信息的线上 `version` 字段使用规范规定的 `2.0`。
- 后端：Go 1.25，优先使用标准库，第三方依赖需有明确收益。
- 部署：单二进制优先，提供 Docker 多架构镜像。
- 数据库：统一支持内存、SQLite、MySQL/MariaDB 和 PostgreSQL，保存信任关系、配置和传输历史。
- 文件访问：发送源和接收目标均限制在各自配置的固定目录内。
- 测试策略：协议组件使用单元测试和互操作集成测试；文件落盘、路径安全和中断恢复作为高风险专项测试。

## 提交策略

- 每个里程碑使用独立提交，提交标题采用 `feat(mN): ...`，其中 `mN` 对应 M0–M7。
- 只有达到该里程碑验收条件并通过格式化、测试和静态检查后，才能创建里程碑提交。
- 未完成的下一阶段代码不混入当前里程碑；紧随其后的修复使用 `fix(mN): ...` 单独提交。
- 文档、迁移和测试随所属功能一同提交，确保任意里程碑提交都可独立构建和验证。

- [x] 支持命令行参数及环境变量配置。
- [x] 启动基础 Web 服务，嵌入占位页面。
- [x] 启动时创建固定发送、接收和数据目录。
- [x] 添加配置及 HTTP 基础测试。
- [x] 记录架构边界、安全基线和后续计划。

验收：`go test ./...` 和 `go vet ./...` 通过；`go run ./cmd/gosend` 可访问 `/healthz`。

### M1：运行基础与持久化

- [x] 引入纯 Go SQLite、MySQL 和 PostgreSQL 驱动及按方言版本化迁移。
- [x] 实现内存、SQLite、MySQL、PostgreSQL 的统一仓储接口。
- [x] 实现设置、信任设备、传输会话、传输文件仓储。
- [x] 生成并持久化 HTTPS 证书；以证书 SHA-256 作为 HTTPS 身份指纹。
- [x] 定义设备、会话、文件状态与错误码等领域模型。
- [x] 增加结构化日志、构建版本和 readiness 状态。

验收：重启后设置和信任记录保留；迁移可从空库重复验证；证书身份稳定；四种数据库通过统一仓储契约测试，其中 MySQL/PostgreSQL 集成测试按测试 DSN 启用。

### M2：LocalSend 设备发现

- [x] 在所有可用 IPv4 网卡监听和发送 `224.0.0.167:53317` UDP 组播公告。
- [x] 实现 `POST /api/localsend/v2/register` 和 `GET /api/localsend/v2/info`。
- [x] 公告触发 HTTPS 注册响应，必要时回退 UDP 响应。
- [x] 维护在线设备、地址、能力及最后在线时间，按 TTL 淘汰。
- [x] 忽略自身指纹，合并重复设备，妥善处理未知设备和协议字段。
- [x] 回连 HTTPS 时按公告指纹验证自签名证书。
- [x] 提供在线设备只读 Web API；默认不主动扫描整个网段。

自动验收：注册/info 端点、设备合并及超时、真实 TLS 进程和多网卡组播启动测试通过。与官方 LocalSend 的多平台人工互操作纳入 M6 矩阵。

### M3：接收与多文件会话

- [x] 实现 `prepare-upload`、`upload` 和 `cancel`。
- [x] 支持自动接受、仅信任设备自动接受、每次手动确认三种策略。
- [x] 一次准备请求创建一个会话和多个独立随机文件令牌。
- [x] 文件流式写入临时目录；校验声明大小和可选 SHA-256 后原子落盘。
- [x] 实现自动追加序号重名策略、磁盘空间/100 GiB 总大小预检、并发冲突、超时和取消清理。
- [x] 记录会话级和文件级结果，允许多文件独立完成或失败。
- [x] 提供待接收请求查询及接受/拒绝 Web API。

自动验收：多文件、手动审批、路径穿越、令牌/IP 绑定、大小/SHA-256、取消和原子无覆盖测试通过。官方客户端互操作纳入 M6 矩阵。

### M4：发送与多文件会话

- [x] 只允许选择固定发送目录内的普通文件，并拒绝越界路径及越界符号链接。
- [x] 获取在线设备能力，调用 `prepare-upload`，再以最大并发 3 上传多个文件。
- [x] 支持 MIME、大小、SHA-256、修改时间等元数据。
- [x] 实现活动文件进度、会话取消和部分失败记录。
- [x] 支持可选 PIN，并将拒绝、忙碌、速率限制和协议状态记录为失败原因。
- [x] 提供启动发送、查询活动进度和取消 Web API。

自动验收：固定目录边界、证书指纹固定、真实 TLS 多文件发送、接收内容和历史状态测试通过。官方客户端与大文件互操作纳入 M6 矩阵。

### M5：Web 管理界面

- [x] 实现响应式仪表盘及运行状态。
- [x] 实现在线设备列表、刷新、地址和身份详情。
- [x] 实现信任设备列表、信任/取消信任操作。
- [x] 实现固定发送目录递归文件浏览器、多选、设备/PIN 和发送操作。
- [x] 实现待接收请求的接受/拒绝交互。
- [x] 实现发送、接收历史以及活动文件进度和取消。
- [x] 使用 SSE 推送发现及传输快照，断线时使用定时重新同步。

自动验收：静态资源嵌入、文件边界、管理 API 与 SSE 编译测试通过；按本地服务约束完成桌面/移动响应式布局。浏览器人工交互回归纳入 M6。

### M6：互操作、安全与可靠性

- [x] 建立官方 LocalSend Android、iOS、Windows、macOS、Linux 人工互操作矩阵。
- [x] 验证 HTTPS 指纹与发现地址绑定，避免设备冒充和降级混淆。
- [x] 增加 JSON 元数据和文件名模糊测试入口。
- [x] 覆盖断网取消、磁盘空间、进程退出、重复会话、慢客户端和并发冲突的状态处理。
- [x] 增加 Web Basic Auth、同源写操作防护和安全响应头。
- [x] 记录数据库、证书和历史记录的备份边界。

自动验收：高风险测试与模糊入口通过；竞态检查加入 M7 Linux CI（当前 Windows 环境没有 race runtime 所需的 C 编译器）。无已知路径越界、任意覆盖、令牌复用或无限资源占用问题。真实官方客户端项目在发布前按 [互操作矩阵](INTEROPERABILITY.md) 人工签核。

### M7：部署与发布

- 提供最小化 Dockerfile、Compose 示例、健康检查和只读根文件系统方案。
- 发布 Linux AMD64/ARM64 二进制，验证 Raspberry Pi 与主流 NAS 容器环境。
- 提供 systemd 服务示例、卷挂载、端口和防火墙文档。
- 加入 CI：格式化检查、测试、`go vet`、漏洞扫描和多架构构建。
- 定义配置升级兼容策略、版本号和发布说明模板。

验收：全新树莓派/NAS 可按文档在 15 分钟内完成部署，并与官方 LocalSend 互传多文件。

## 建议实施顺序

严格按 M1 → M2 → M3 → M4 → M5 推进。Web 界面在 M5 集中完善，但每个前置里程碑先提供最小内部 API/测试工具，保证协议和传输能力可独立验证。M6 的安全测试不应等到最后才开始：每个传输端点实现时同步加入对应攻击面测试。

## 首批待定决策

数据库驱动与多后端策略已记录在 [ADR 0001](adr/0001-multi-database-store.md)。剩余选择应在对应里程碑开始前确定并记录 ADR：

1. Web 前端采用原生轻量方案还是独立 TypeScript 构建。
2. 默认接收策略是手动确认还是仅信任设备自动接收。
3. 重名文件采用拒绝、覆盖还是自动追加序号。
4. 是否在首个正式版本提供 HTTP 兼容模式，以及该模式的 UI 风险提示。
