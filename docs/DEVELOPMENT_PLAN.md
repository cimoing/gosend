# GoSend 开发计划

## 基线与原则

- 协议基线：LocalSend Protocol v2.1；设备信息的线上 `version` 字段使用规范规定的 `2.0`。
- 后端：Go 1.25，优先使用标准库，第三方依赖需有明确收益。
- 部署：单二进制优先，提供 Docker 多架构镜像。
- 数据库：SQLite，保存信任关系、配置和传输历史。
- 文件访问：发送源和接收目标均限制在各自配置的固定目录内。
- 测试策略：协议组件使用单元测试和互操作集成测试；文件落盘、路径安全和中断恢复作为高风险专项测试。

## 提交策略

- 每个里程碑使用独立提交，提交标题采用 `feat(mN): ...`，其中 `mN` 对应 M0–M7。
- 只有达到该里程碑验收条件并通过格式化、测试和静态检查后，才能创建里程碑提交。
- 未完成的下一阶段代码不混入当前里程碑；紧随其后的修复使用 `fix(mN): ...` 单独提交。
- 文档、迁移和测试随所属功能一同提交，确保任意里程碑提交都可独立构建和验证。

## 里程碑

### M0：项目初始化（已完成）

- [x] 建立 Go module、程序入口和内部包结构。
- [x] 支持命令行参数及环境变量配置。
- [x] 启动基础 Web 服务，嵌入占位页面。
- [x] 启动时创建固定发送、接收和数据目录。
- [x] 添加配置及 HTTP 基础测试。
- [x] 记录架构边界、安全基线和后续计划。

验收：`go test ./...` 和 `go vet ./...` 通过；`go run ./cmd/gosend` 可访问 `/healthz`。

### M1：运行基础与持久化

- 引入 SQLite 驱动和版本化迁移。
- 实现设置、信任设备、传输会话、传输文件仓储。
- 生成并持久化 HTTPS 证书；以证书 SHA-256 作为 HTTPS 身份指纹。
- 定义设备、会话、文件状态与错误码等领域模型。
- 增加结构化日志、构建版本和 readiness 状态。

验收：重启后设置和信任记录保留；迁移可从空库重复验证；证书身份稳定。

### M2：LocalSend 设备发现

- 监听和发送 `224.0.0.167:53317` UDP 组播公告。
- 实现 `POST /api/localsend/v2/register` 和 `GET /api/localsend/v2/info`。
- 公告触发 HTTP 注册响应，必要时回退 UDP 响应。
- 维护在线设备、地址、能力及最后在线时间，按 TTL 淘汰。
- 忽略自身指纹，合并重复设备，妥善处理未知设备类型和协议小版本。
- 为不支持组播的环境预留手动注册/子网扫描接口，但默认不主动扫描整个网段。

验收：与至少两个官方 LocalSend 平台双向发现；断线设备能在预期时间内离线。

### M3：接收与多文件会话

- 实现 `prepare-upload`、`upload` 和 `cancel`。
- 支持自动接受、仅信任设备自动接受、每次手动确认三种策略。
- 一次准备请求创建一个会话和多个文件令牌。
- 文件流式写入临时目录；校验声明大小和可选 SHA-256 后原子落盘。
- 实现重名策略、空间预检、并发/总大小限制、超时和取消清理。
- 记录会话级和文件级结果，允许部分成功。

验收：官方 LocalSend 可一次发送多个文件；拒绝路径穿越、超长请求、错误令牌、大小不符和越界符号链接。

### M4：发送与多文件会话

- 只列出并选择固定发送目录内的普通文件。
- 获取在线设备能力，调用 `prepare-upload`，再受控并发上传多个文件。
- 支持 MIME、大小、SHA-256、修改时间等元数据。
- 实现进度、取消、重试和部分失败展示。
- 处理 PIN、拒绝、忙碌、速率限制和协议错误。

验收：一次选择多个文件可发送给官方 LocalSend；取消后双方会话可释放；大文件传输保持稳定内存占用。

### M5：Web 管理界面

- 仪表盘及运行状态。
- 在线设备列表、手动刷新和设备详情。
- 信任设备列表、信任/取消信任操作。
- 固定发送目录文件浏览器、多选和发送确认。
- 待接收请求的接受/拒绝交互。
- 发送、接收历史以及会话/文件级状态和进度。
- 使用 SSE 推送发现及传输进度，断线后可重新同步快照。

验收：桌面与移动浏览器可完成所有主要操作；Web API 有权限、输入和并发验证。

### M6：互操作、安全与可靠性

- 建立官方 LocalSend Android、Windows/macOS/Linux 的互操作矩阵。
- 验证 HTTPS 指纹与发现地址绑定，避免设备冒充和降级混淆。
- 模糊测试 JSON 协议解析、查询参数和文件名处理。
- 覆盖断网、磁盘满、进程退出、重复包、慢客户端和并发冲突。
- 增加 Web 管理面认证选项、CSRF 防护和安全响应头。
- 制定数据库、证书和历史记录的备份/恢复方案。

验收：高风险测试全部自动化；无已知路径越界、任意覆盖、令牌复用或无限资源占用问题。

### M7：部署与发布

- 提供最小化 Dockerfile、Compose 示例、健康检查和只读根文件系统方案。
- 发布 Linux AMD64/ARM64 二进制，验证 Raspberry Pi 与主流 NAS 容器环境。
- 提供 systemd 服务示例、卷挂载、端口和防火墙文档。
- 加入 CI：格式化检查、测试、`go vet`、漏洞扫描和多架构构建。
- 定义配置升级兼容策略、版本号和发布说明模板。

验收：全新树莓派/NAS 可按文档在 15 分钟内完成部署，并与官方 LocalSend 互传多文件。

## 建议实施顺序

严格按 M1 → M2 → M3 → M4 → M5 推进。Web 界面在 M5 集中完善，但每个前置里程碑先提供最小内部 API/测试工具，保证协议和传输能力可独立验证。M6 的安全测试不应等到最后才开始：每个传输端点实现时同步加入对应攻击面测试。

## 首批待定决策

以下选择不阻塞 M0，但应在 M1 开始前确定并记录 ADR：

1. SQLite 驱动采用纯 Go 实现还是 CGO 实现。
2. Web 前端采用原生轻量方案还是独立 TypeScript 构建。
3. 默认接收策略是手动确认还是仅信任设备自动接收。
4. 重名文件采用拒绝、覆盖还是自动追加序号。
5. 是否在首个正式版本提供 HTTP 兼容模式，以及该模式的 UI 风险提示。
